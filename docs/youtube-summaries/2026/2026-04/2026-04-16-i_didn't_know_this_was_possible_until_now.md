## 주요 목차별 상세 요약

### [00:00] MCP 컨텍스트 오염 문제 제기
- MCP의 본질적 문제를 `context bloat`로 설정
- 도구 설명과 출력이 컨텍스트 윈도우를 불필요하게 점유
- 기존 대안들은 문제를 인식했지만 완전한 해결책은 아님
  - 근거와 예시
    - Docker `code mode`: 도구를 JS 코드로 직접 호출해 노출 문제를 줄이지만 Docker MCP에 종속
    - Cloudflare 제안: 도구를 컨텍스트 항목이 아니라 실행 가능한 코드로 다루는 방향
    - Anthropic도 이 갭을 인정해 관련 논문 발표
    - TypeScript 파일화 방식: MCP가 많을수록 변환·검증 부담 급증
    - Claude Code CLI 플래그: 도구 노출은 늦추지만 출력 잔류 문제는 남음
    - CLI hub류: 빌드 타임 변환 중심이라 유지보수 비효율
  - 세부 요약
    - 문제의 초점은 단순히 도구 수가 아니라, 컨텍스트를 얼마나 오염시키는가
    - 핵심 전제는 에이전트가 도구를 보기는 해야 하지만, 매번 전부 들고 있을 필요는 없다는 점
    - 기존 방식들은 노출 억제 또는 코드 변환 중 한쪽만 해결

### [01:30] MCP2 CLI: 서버를 bash 도구로 변환
- MCP 서버를 런타임에 CLI/bash 명령으로 바꾸는 접근
- 도구 출력은 필요할 때만 에이전트 컨텍스트에 주입
- 원본 MCP 갱신이 변환된 도구에도 자동 반영되는 구조
  - 근거와 예시
    - MCP Python SDK 위에 구축
    - 실제 호출 시점에 bash 명령으로 실행
    - 원격/로컬 MCP와 범용 API를 같은 CLI 인터페이스로 다룸
    - OpenAPI, REST API도 동일한 방식으로 연결 가능
    - 설치형 `pip` 방식과 비설치 실행 모두 지원
    - 팀은 작업 환경 오염을 피하려고 비설치 실행을 선택
  - 세부 요약
    - 핵심은 빌드 타임이 아니라 런타임 변환
    - 도구 정의를 코드로 복제하지 않고, 필요 시점에 실행 경로만 열어둠
    - 실행 후 결과는 자동으로 계속 쌓이지 않고 요청에 따라 들어옴

### [03:00] 캐시와 토큰/속도 최적화
- 자주 쓰는 도구는 캐시에 넣어 재조회 비용을 줄임
- 캐시와 런타임 유연성 사이의 절충
- 목표는 토큰 절약과 빠른 실행
  - 근거와 예시
    - 캐시 TTL 1시간
    - frequently used MCP tools를 캐시에 저장
    - 빠른 retrieval로 재사용
    - Tik Token으로 자동화 테스트
    - 토큰 사용량이 더 적고 실행도 더 빠르다고 제시
  - 세부 요약
    - 도구를 항상 새로 해석하지 않고, 자주 쓰는 것만 남겨 성능을 확보
    - 도구 출력이 크더라도 그 자체를 계속 컨텍스트에 들고 있지 않게 설계
    - 런타임 변환과 캐시를 같이 써서 유지보수와 속도를 동시에 겨냥

### [04:30] Orchids: 환경에서 직접 앱 빌드·배포
- 환경에서 바로 앱을 만들고 배포하는 에이전트 데모
- 여러 스택을 가리지 않고 동작하는 방향
- 외부 구독을 직접 가져와 쓰는 흐름
  - 근거와 예시
    - first AI agent that can build and deploy any app on any stack directly from your environment
    - bring your own subscription 언급
    - 프로젝트 생성과 연결 코드 추가까지 수행
    - Superbase MCP, GitHub MCP, Puppeteer MCP, Context 7 MCP 연결 사례
  - 세부 요약
    - 핵심 메시지는 단순 연결이 아니라, 실제 작업 흐름까지 이어져야 한다는 점
    - 에이전트가 도구를 읽는 것과 도구를 실제로 쓰는 것은 다른 문제
    - 데모는 MCP2 CLI가 그 간극을 줄이는 방향임을 보여주는 장면

### [06:00] bash 실행의 보안 우려
- bash 명령으로 실행되면 민감정보 노출 우려가 생김
- API 키와 액세스 토큰을 명령행 인자로 두는 방식은 위험
- 명령행 외 경로로 비밀을 처리해야 한다는 전제
  - 근거와 예시
    - 프로세스 목록에 노출될 수 있다는 걱정 제기
    - API keys, access tokens 노출 가능성
    - 명령행 인자 대신 환경변수 사용
    - 파일 경로와 secret manager로 처리
  - 세부 요약
    - 도구를 CLI화하면 단순 실행성은 좋아지지만 비밀 관리가 새 쟁점이 됨
    - 문제는 CLI 자체가 아니라, 비밀을 어디에 실어 보내느냐
    - 안전한 운용을 위해 명령 인자 노출을 피하는 쪽으로 설계

### [07:30] 도구 연결만으로는 사용 유도가 안 됨
- 연결만으로는 에이전트가 적절한 시점에 도구를 쓰지 않음
- `claude.md`보다 `skill`이 더 직접적인 보정 수단
- 대형 출력은 형식 제어와 파일 분리가 필요
  - 근거와 예시
    - `claude.md`에 "코드 작성 전 context 7 MCP를 먼저 쓰라"는 지시 추가
    - deprecated `middleware file` 대신 최신 `Next.js`에서는 `proxy`를 사용해야 한다는 문서 수정
    - `context 7 MCP`로 문서를 불러와 deprecated 설정을 교정
    - `skill`에는 보유 도구, 사용법, 언제 써야 하는지 포함
    - `tune format`: indentation + CSV 스타일을 섞어 JSON/YAML보다 압축적
  - 세부 요약
    - 핵심은 도구 연결보다 도구 사용 규칙을 에이전트가 읽을 수 있게 만드는 것
    - `claude.md`는 임시 보정책이고, `skill`이 더 체계적
    - 큰 출력은 컨텍스트에 덤프하지 말고 파일로 빼는 쪽이 실용적

### [09:00] 브라우저 기반 테스트의 세션 한계
- 채팅 기능 테스트에서 실제 세션 유지가 깨짐
- Puppeteer MCP로 직접 클릭·스크롤·상호작용하는 검증이 유용
- 브라우저 인스턴스가 매 호출마다 새로 떠 세션을 못 유지
  - 근거와 예시
    - 테스트용으로 사용자 2명 생성
    - 메시지 로딩 실패, 로딩 화면만 표시
    - tool call마다 새 browser instance 생성
    - 세션 데이터 유지 실패
    - 단순 작업인데도 7분 낭비 언급
  - 세부 요약
    - 정적 코드 리뷰만으로는 놓치기 쉬운 실제 플로우 문제가 드러남
    - 하지만 브라우저 세션을 계속 살릴 수 없어서 효율이 떨어짐
    - 이 제한이 CLI 방식의 필요성을 더 강화

### [10:30] 파일 기반 출력 처리와 실전 적용
- MCP 결과를 컨텍스트가 아니라 파일처럼 다루는 방식이 전환점
- Grep으로 추출하고, `progress.json`으로 진행 상태를 관리
- 네이티브 MCP 처리에서 안 되던 워크플로가 CLI로 가능해짐
  - 근거와 예시
    - MCP 결과를 컨텍스트에 덤프하지 않고 파일로 파이프
    - Grep로 패턴 매칭해 필요한 데이터만 추출
    - `progress.json`으로 프로젝트 진행 추적
    - 총 78개 도구를 확인
    - Claude가 여러 MCP를 실제 작업에 연결한 사례 제시
  - 세부 요약
    - 핵심은 대형 출력을 그대로 읽히지 않게 하고, 필요한 부분만 뽑아 쓰는 것
    - 에이전트 네이티브 MCP 처리에서는 한계가 있었지만 CLI로 우회 가능
    - 영상의 결론은 도구를 더 많이 붙이는 것보다, 도구를 다루는 방식이 더 중요하다는 것

## 핵심 결론
- 최종 메시지는 MCP의 성패가 연결 수가 아니라 컨텍스트 제어, 사용 규칙, 출력 처리 방식에 달려 있다는 점
- MCP2 CLI는 도구를 CLI로 옮겨 에이전트가 실제로 쓰게 만들고, 출력은 필요할 때만 들여오게 하려는 해법
