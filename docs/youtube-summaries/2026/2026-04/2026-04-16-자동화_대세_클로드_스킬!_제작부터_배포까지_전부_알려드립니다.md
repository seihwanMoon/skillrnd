## 주요 목차별 상세 요약

### [00:00] 결과 편차 문제와 견적서 자동화 필요
- 같은 요청을 반복해도 결과물이 매번 달라지는 문제를 출발점으로 설정
- 회사 정보 누락, 형식 붕괴, 재작성 비용을 줄이려는 목적
- 예시 주제는 견적서 초안 자동 생성
  - 근거와 예시
    - 고객용 요청을 다시 만들었는데 결과가 달라지는 경험
    - 비활용 시 입금 계좌 정보 누락, 부가세 표기 불일치 같은 오류
    - 비활용 출력에서 정렬 불안정, 레이아웃 붕괴 사례
  - 세부 요약
    - 핵심은 '더 좋은 한 번'보다 '같은 품질의 반복' 확보
    - 프롬프트 단발성 해결이 아니라 재현성 문제로 문제 정의

### [01:30] 클로드 스킬의 역할 정의
- `클로드 스킬`을 AI용 SOP, 업무 매뉴얼처럼 해석
- `description`은 트리거, `skill.md`는 실행 절차와 출력 규칙
- 신입사원 온보딩용 매뉴얼, n8n 워크플로우와 유사
  - 근거와 예시
    - 트리거 -> 액션 -> 최종 아웃풋의 흐름으로 설명
    - 3레이어 구조: 이름/설명, 본문, 참고 문서·코드
    - 단순 프롬프트보다 전체 워크플로우를 고정
  - 세부 요약
    - 1층은 항상 로딩, 2층은 실제 지침, 3층은 필요 시 불러오는 무거운 자원
    - 텍스트뿐 아니라 코드, 문서, 서브 에이전트까지 연결 가능

### [03:00] 3레이어 구조와 토큰 효율
- 항상 로딩되는 정보는 최소화, 필요 자원은 지연 로딩
- 재사용성, 일관성, 확장성 확보
- 토큰 절약과 표준화된 산출물 생산이 핵심
  - 근거와 예시
    - 설명 분량은 약 80토큰 수준으로 먼저 잡힘
    - 참고 문서와 스크립트는 필요할 때만 로딩
    - 반복 복붙 대신 스킬 트리거만으로 실행
  - 세부 요약
    - 무거운 PDF, 코드, 체크리스트를 분리해 관리
    - 출력 포맷이 흔들리지 않도록 체크리스트와 규칙을 묶음

### [06:00] 프롬프트 대비 적용 경계
- 간단한 1~2회 지시는 여전히 프롬프트로 충분
- 반복 자동화와 일정한 형식 유지가 필요한 업무에 스킬 적합
- 초기 설계와 프로세스 정의가 약하면 효용이 떨어짐
  - 근거와 예시
    - 스킬 없이도 간단한 요청은 처리 가능하다는 반론 제시
    - 프로세스 정의가 부족하면 만족할 만한 결과물 나오지 않음
    - 템플릿·레퍼런스 품질이 출력 품질을 좌우
  - 세부 요약
    - 스킬은 만능 대체재가 아니라 반복·표준화용 도구
    - 구조가 정리되지 않으면 자동화해도 결과가 흔들림

### [07:30] 제작 환경과 플러그인 설치
- 제작 환경은 VS Code 터미널 + Claude Code
- `Skill Creator` 플러그인 설치로 스킬 생성 시작
- 승인 흐름을 줄이는 옵션도 사용
  - 근거와 예시
    - VS Code 메뉴에서 터미널 열기
    - 플러그인 설치 후 슬래시 명령으로 스킬 호출
    - 손쉽게 스킬을 가져와 활용하는 흐름
  - 세부 요약
    - 터미널 기반이라 빠르게 수정과 재실행 가능
    - 스킬 제작 도구를 먼저 깔아야 생성 작업이 진행

### [09:00] 레퍼런스 기반 스킬 설계
- 인보이스 템플릿 PDF를 `reference` 폴더에 넣어 기준 구조 확보
- 고객사명, 프로젝트 내용, 업무 항목, 단가를 입력받아 견적 초안 생성
- 출력 형식은 PDF, 자체 검증 체크리스트 포함
  - 근거와 예시
    - 템플릿 구조를 반영해 생성 결과가 안내됨
    - 특정 요청이 들어오면 자동 트리거되도록 설계
    - 체크리스트 항목이 결과물 구조 점검 기준
  - 세부 요약
    - 정렬과 레이아웃 기준을 템플릿에서 끌어옴
    - 스킬이 어떤 요청에서 발동할지 명시해야 함

### [12:00] AB 테스트와 검증 루프
- 스킬 버전 3개와 미사용 버전 3개, 총 6개 병렬 AB 테스트
- 이밸루에이션과 그레이딩으로 결과 비교
- 체크리스트 통과 여부가 성능 판단의 기준
  - 근거와 예시
    - 스킬 사용 버전은 기본 구조를 거의 그대로 반영
    - 포맷그레이드에서 체크리스트 항목이 모두 체크
    - 스킬 활용 약 34초, 비활용 약 147초(+/- 60초)
  - 세부 요약
    - 비교 대상은 스킬 사용/미사용의 동일 인풋
    - 정량 비교 후 수정 포인트를 잡는 흐름

### [18:00] 피드백 반영과 스타일 정렬
- 피드백을 반영해 `skill.md`를 업데이트
- 브랜드 가이드 문서 추가로 색상, 폰트, 헤더/푸터 정렬
- 이전 출력과 비교하며 벤치마크 개선
  - 근거와 예시
    - 브랜드 가이드 MD 추가 후 주황색 메인 컬러로 통일
    - `Previous Output`와 비교 가능한 구조
    - 입금 계좌 정보 누락, 부가세 표기 불일치 같은 오류 수정 포인트
  - 세부 요약
    - 피드백은 자동 저장된 로컬 폴더 기준으로 이어짐
    - 가이드 문서가 있어야 미적 정렬이 안정화

### [22:30] 플러그인·마켓플레이스 배포
- 스킬 폴더를 플러그인 형태로 패키징
- `claude` 대시 플러그인 구조와 `marketplace.json` 구조로 분기
- zip 파일로 묶어 배포, Claude Code와 Claude COWORK에서 활용
  - 근거와 예시
    - `Add Marketplace`로 마켓플레이스 로드
    - 로컬 폴더 복사만으로 쓰는 방식과 배포형 플러그인 구분
    - 제작 완료 후 앱에 넣어 쓰는 흐름
  - 세부 요약
    - 플러그인 생성 시 마켓플레이스 관련 파일과 zip 파일 패키징
    - 배포 후 다른 프로젝트, 다른 환경에서 재사용

### [25:30] 스킬 없는 실행의 한계
- 스킬이 없으면 레이아웃, 디자인 의도, 프로세스가 불명확
- 결과물은 정렬 불안정, 종이 뜯긴 듯한 레이아웃으로 흐를 수 있음
- 반복 업무 자동화는 결국 스킬 설계와 프로세스 정의 문제
  - 근거와 예시
    - 스킬 미사용 시 회사 정보, 형식, 부가세 표기 등 누락 사례
    - 프로세스가 제대로 정의되지 않으면 만족도 낮은 결과
    - 브랜드 가이드가 없으면 색상·폰트 정렬 약화
  - 세부 요약
    - 스킬은 단발성 응답 보정이 아니라 반복 작업의 기준화
    - 초기 설계 품질이 결과 품질을 좌우

## 핵심 결론
- 반복 업무를 AI에 맡기려면 프롬프트만 다듬는 방식보다, 실행 절차·템플릿·검증·배포를 묶은 스킬로 고정하는 쪽이 유효
- 스킬은 결과를 한 번 잘 뽑는 도구가 아니라, 같은 품질을 계속 재현하게 만드는 장치
